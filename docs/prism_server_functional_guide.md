# PRISM Server 功能说明

## 1. 产品定位

PRISM Server 是一个本地交互式分析面板，用来围绕单细胞数据集和可选模型结果做快速浏览、校验和单基因分析。

它面向的是研究、调试和解释性探索场景，而不是面向公网部署的多用户平台。

## 2. 目标用户

适合以下人群：

- 需要快速检查本地 `.h5ad` 数据集的研究人员
- 需要验证模型结果是否可用的开发者或算法工程师
- 需要围绕单个基因做交互式探索的人
- 需要用浏览器完成轻量分析而不是写脚本的人

## 3. 核心能力总览

当前模块已经具备以下功能能力：

1. 加载本地数据集
2. 可选加载模型结果文件
3. 展示数据集摘要信息
4. 展示模型结果摘要信息
5. 浏览和搜索基因
6. 进入单基因详情页
7. 查看原始表达统计
8. 基于模型结果做单基因 posterior 分析
9. 在页面内触发单基因即时拟合
10. 做按分组的 kBulk 抽样分析
11. 输出少量 JSON 接口用于健康检查和搜索

## 4. 使用方式

### 4.1 启动后的基本流程

用户进入页面后，典型使用顺序如下：

1. 输入数据集路径
2. 视需要输入模型结果路径和表达层名称
3. 执行加载
4. 在首页查看数据与模型摘要
5. 在基因浏览区域筛选目标基因
6. 打开某个基因的详情页
7. 在详情页切换分析模式、prior 来源、标签或分析参数

### 4.2 这是一个单上下文面板

系统一次只维护一份当前分析上下文。

这意味着：

- 新的加载操作会替换当前数据上下文
- 新上下文会覆盖旧上下文
- 缓存会跟随当前上下文切换
- 更适合本地单用户使用，而不是多人共享同一实例

## 5. 数据加载能力

### 5.1 支持的输入

页面支持输入三类信息：

- 必填的数据集路径
- 可选的模型结果路径
- 可选的表达层名称

### 5.2 加载成功后会提供的能力

成功加载后，用户可以立刻获得：

- 细胞数
- 基因数
- 总计数分布概览
- 可用标签字段
- 基因浏览能力
- 若提供模型结果，则同时获得模型摘要和 checkpoint-backed 分析能力

### 5.3 标签识别

如果数据中存在适合做分组分析的标签字段，页面会把它们暴露出来，用于：

- 标签 prior 分析
- 标签过滤后的基因分析
- kBulk 分组比较

如果没有合适的标签字段，系统仍然可以提供基础的基因浏览和 raw 分析，但分组能力会下降。

## 6. 首页能力

首页承担的是“加载入口 + 全局概览 + 基因入口”的角色。

### 6.1 未加载状态

在未加载任何数据时，首页主要提供：

- 数据加载表单
- 对系统用途的简要说明
- 对可执行分析类型的概览说明

### 6.2 已加载状态

在加载成功后，首页会扩展为一个总览面板，展示：

- 数据集快照
- 模型摘要
- 标签信息
- 当前使用的数据源路径
- 基因浏览器

### 6.3 数据集快照

数据集快照强调的是“快速判断这个数据是否符合预期”。

常见展示内容包括：

- 细胞数
- 基因数
- 当前表达层
- 总 count 的统计概览
- 可用标签字段
- 当前数据路径

### 6.4 模型摘要

如果加载了模型结果，首页会展示模型的关键信息，帮助用户快速回答这些问题：

- 是否存在 global prior
- 是否存在标签 prior
- 覆盖了多少基因
- 使用的分布类型是什么
- support 属于什么语义空间
- scale 和 reference 信息是否可用
- 与当前数据是否有足够的 reference overlap
- 推荐使用哪个标签字段

### 6.5 基因浏览器

首页提供一个 gene browser，用来把“全局概览”自然过渡到“单基因分析”。

它支持：

- 基因名子串搜索
- 按表达量排序
- 按检测细胞数排序
- 按检测比例排序
- 按基因名排序
- 按基因索引排序
- 升序和降序
- 分页
- 根据是否存在模型结果决定浏览范围

## 7. 基因搜索能力

系统同时提供两类搜索体验。

### 7.1 浏览式搜索

浏览式搜索用于在首页的基因表格中缩小候选范围，适合模糊查找。

### 7.2 直接定位

直接定位用于进入基因详情页，适合已知目标基因时快速跳转。

它支持以下定位方式：

- 精确基因名
- 数字索引
- 大小写不敏感的精确匹配

## 8. 基因详情页能力

基因详情页是系统的核心页面。

它把多个能力聚合在同一个工作区中：

- 分析模式切换
- prior 来源切换
- 标签选择
- 单基因 raw 统计
- posterior 结果概览
- 图形化诊断
- on-demand fit
- kBulk 分析

## 9. 三种分析模式

### 9.1 Raw only

这是最基础的模式，不依赖模型结果。

它用于回答：

- 这个基因在当前数据中的原始表达是什么样
- 计数分布是否稀疏
- 检测比例高不高
- 与细胞总深度的关系如何

### 9.2 Checkpoint posterior

如果当前基因在模型结果中可用，系统可以直接用现成 prior 做 posterior 分析。

这个模式用于回答：

- 当前基因在模型 prior 下的 signal 如何
- posterior 不确定性如何
- 信息量如何
- 不同标签 prior 是否改变结果

### 9.3 On-demand fit

如果用户希望不直接使用现有 prior，而是对当前基因即时重新拟合，系统支持在页面中直接触发拟合。

这个模式用于回答：

- 新拟合 prior 与已有 prior 是否一致
- 拟合过程是否稳定
- 新 prior 下的 posterior 结果是否更合理

## 10. Prior 来源能力

系统支持两类 prior 来源：

- global
- label

### 10.1 Global

global 表示用统一 prior 处理当前分析对象。

### 10.2 Label

label 表示基于某个分组标签选择特定 prior，并且只对该标签对应的细胞子集做分析。

这让页面既能做“全局解释”，也能做“特定群体解释”。

## 11. 原始统计能力

对于任意基因，系统都能给出基础原始统计，用来帮助用户快速判断表达形态。

典型统计内容包括：

- 总 count
- 均值
- 中位数
- 高分位数
- 最大值
- 检测细胞数
- 检测比例
- 零值比例
- 与细胞总 count 的相关关系

## 12. 图形化诊断能力

基因详情页会配合图形展示不同维度的信息。

当前能力覆盖：

- 原始表达概览图
- prior 对比图
- posterior signal 诊断图
- posterior gallery
- 拟合目标变化曲线
- kBulk 分组比较图

这些图的目标不是追求出版级排版，而是用于帮助用户快速判断结果是否可信、是否异常、是否值得进一步导出到正式分析流程。

## 13. On-demand fit 能力

页面提供一组拟合参数入口，让用户在浏览器里直接尝试不同拟合配置。

这部分能力适合以下使用场景：

- 想快速验证某个基因是否适合即时拟合
- 想比较不同 reference 策略
- 想调整拟合分辨率和收敛行为
- 想观察新 prior 与原 checkpoint prior 的差异

## 14. kBulk 能力

系统支持围绕单个基因做按类分组的 kBulk 抽样分析。

它的目标是提供一种轻量、交互式的群组层面比较视角，用来回答：

- 不同标签组的 signal 水平是否明显不同
- 不同标签组的不确定性是否存在差异
- 在给定抽样规模下结果是否稳定

页面会展示：

- 所用分组键
- 所用 prior 来源
- 每组参与的细胞数
- 每组实际抽样次数
- 每组 signal 汇总
- 每组 entropy 汇总
- 分组对比图

## 15. API 能力

除了 HTML 页面，系统还保留了少量 JSON 接口。

### 15.1 健康检查

健康检查接口用于确认服务是否正常，以及当前是否已经加载了数据。

### 15.2 基因搜索

搜索接口用于给前端或外部脚本提供轻量基因候选结果，适合做自动补全或快速查找。

## 16. 错误展示原则

系统的错误策略偏向“直接在页面上可见”，而不是隐藏在日志里。

常见错误场景包括：

- 路径为空
- 数据加载失败
- 模型结果不兼容
- 基因不存在
- 目标基因没有可用 prior
- 标签选择无效
- kBulk 条件不满足

在这些情况下，系统通常会：

- 在页面上直接显示错误信息
- 尽量保留用户之前输入的关键参数
- 在可能时回退到更基础的分析模式，而不是让整页不可用

## 17. 性能与运行边界

### 17.1 共享状态

系统以单实例共享状态运行，不做多用户隔离。

### 17.2 本地文件导向

系统依赖本地文件路径，不负责文件上传。

### 17.3 面向交互式探索

它更适合：

- 本地调试
- 小团队内部使用
- 面向单个数据上下文的分析会话

它不适合：

- 多租户平台化部署
- 长任务异步队列管理
- 权限隔离和审计场景

## 18. 非目标

当前模块不追求以下方向：

- 用户系统
- 权限控制
- 文件上传
- 数据持久化任务编排
- 面向外部平台的完整 REST 服务
- 多用户会话隔离
- 模型结果写回或编辑

## 19. 当前产品价值

这个模块的价值不在于“替代正式分析流水线”，而在于把常见的局部分析动作压缩进一个低门槛、本地可用、浏览器可交互的工作面板中。

它适合作为：

- 数据集与模型结果的快速质检入口
- 单基因问题排查工具
- prior / posterior 解释界面
- on-demand fit 与 kBulk 的实验台

## 20. 后续重构方向

如果继续演进，这个模块最值得强化的方向包括：

1. 更清晰的页面信息层级
2. 更稳定的移动端与窄屏体验
3. 更一致的交互反馈
4. 更清晰的错误恢复路径
5. 更强的分析工作流连续性

本文件刻意忽略实现细节，只保留用户视角、产品能力和运行边界。
